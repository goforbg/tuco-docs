## Overview

Message webhooks push status updates to a URL you control whenever a message
reaches a terminal state. Configure them in your workspace settings â€” Tuco
handles delivery automatically.

<Info>
Webhook calls are **POST** requests with a JSON body. All events share the same
base fields; each event type adds its own specific fields.
</Info>

## Event types

| Event | When | Status field |
|-------|------|-------------|
| `message.delivered` | Message confirmed delivered to recipient's device | `"delivered"` |
| `message.failed` | Message failed after all retries (technical error) | `"failed"` |
| `message.fallback` | Recipient has no iMessage; primary channel unavailable | `"fallback"` |
| `message.reply` | A lead or contact replied to a message you sent | â€” |

## `message.reply`

Sent when a new reply from a lead or contact is recorded. Use it to update your CRM, trigger automations, or log conversations.

**Payload shape (backward compatible):** Existing fields are unchanged. New optional fields are additive so older integrations keep working.

| Field | Description |
|-------|-------------|
| `event` | `"message.reply"` |
| `timestamp` | When the webhook was sent (ISO UTC) |
| `workspaceId` | Workspace that owns the conversation |
| `messageId` | ID of the **reply** message (the one they sent) |
| `message` | Full reply message object (body, recipient, etc.) |
| `originalMessageId` | ID of the message they replied to (when known) |
| `originalMessage` | Original outbound message object (when known) |
| `leadId` | Lead ID (when the reply is linked to a lead) |
| `lead` | Full lead object (when known) |
| `campaignId` / `campaignName` | Set when the reply is tied to a campaign |
| **`repliedAtUtc`** | **(New)** When the reply was received, in UTC (ISO 8601). Always present. |
| **`parentMessages`** | **(New)** Last 2 outbound messages you sent to this lead before the reply. Each has `messageId`, `message` (snippet), `sentAtUtc`, and optionally `batchId` / `stepIndex`. Omitted when there are none. |

**When the reply is outside a campaign:**

- **`repliedAtUtc`** â€” Always present. Every reply has a received-at time.
- **`parentMessages`** â€” When the reply is not tied to a campaign, Tuco still includes the last 2 outbound messages you sent to that lead in the workspace (any channel). So you always get context about what you last sent them, even for one-off or non-campaign replies. When the reply is tied to a campaign, `parentMessages` is scoped to that campaignâ€™s outbound messages.

Example payload (new fields are additive; older subscribers can ignore them):

```json
{
  "event": "message.reply",
  "timestamp": "2025-03-02T15:00:00.000Z",
  "workspaceId": "org_xxx",
  "messageId": "674abc...",
  "message": { "message": "Thanks, I'm interested!", "direction": "received" },
  "originalMessageId": "674def...",
  "leadId": "667f1f77bcf86cd799439012",
  "lead": { "firstName": "Jane", "lastName": "Doe" },
  "campaignId": "674...",
  "campaignName": "Q1 Outreach",
  "repliedAtUtc": "2025-03-02T14:59:58.000Z",
  "parentMessages": [
    { "messageId": "674def...", "message": "Hi Jane, we have an offer...", "sentAtUtc": "2025-03-01T10:00:00.000Z", "batchId": "campaign_674..._0", "stepIndex": 0 }
  ]
}
```

## `message.failed`

Sent when a message has exhausted all retry attempts or encountered an
unrecoverable error (e.g. availability API failure).

```json
{
  "event": "message.failed",
  "messageId": "690d1c95984d3cdcac76e9ff",
  "status": "failed",
  "message": "Hello from Tuco đź‘‹",
  "messageType": "imessage",
  "recipientPhone": "+12025551234",
  "recipientName": "Jane Doe",
  "leadId": "667f1f77bcf86cd799439012",
  "workspaceId": "org_123",
  "createdByUserId": "user_123",
  "reason": "Send failed after 5 attempts",
  "timestamp": "2025-10-15T14:35:00.000Z"
}
```

<ResponseField name="event" type="string">`"message.failed"`</ResponseField>
<ResponseField name="reason" type="string">Human-readable explanation of the failure</ResponseField>

<Tip>
When `sendFallbackSmsOnFailed` is enabled (per-message or workspace-level) and
Twilio is configured, Tuco automatically sends a fallback SMS on this event.
You don't need to build that logic yourself.
</Tip>

## `message.fallback`

Sent when Tuco determines that the primary channel (e.g. iMessage) is not
available for this recipient. This is a **business outcome**, not a technical
failure.

```json
{
  "event": "message.fallback",
  "messageId": "690d1c95984d3cdcac76e9ff",
  "status": "fallback",
  "message": "Hello from Tuco đź‘‹",
  "messageType": "imessage",
  "recipientEmail": "user@example.com",
  "recipientPhone": "+12025551234",
  "recipientName": "Jane Doe",
  "leadId": "667f1f77bcf86cd799439012",
  "workspaceId": "org_123",
  "createdByUserId": "user_123",
  "reason": "Prospect's iMessage not available on +12025551234, user@example.com",
  "checkedAddresses": [
    "+12025551234",
    "user@example.com"
  ],
  "timestamp": "2025-11-06T22:10:05.864Z"
}
```

<ResponseField name="event" type="string">`"message.fallback"`</ResponseField>
<ResponseField name="reason" type="string">Human-readable explanation</ResponseField>
<ResponseField name="checkedAddresses" type="string[]">
  All addresses Tuco checked for iMessage availability before deciding fallback.
  Useful for debugging and auditing.
</ResponseField>

## Shared fields (all events)

Every webhook payload includes these base fields:

| Field | Description |
|-------|-------------|
| `event` | Event type (`message.delivered`, `message.failed`, `message.fallback`) |
| `messageId` | Tuco message ID â€” use as your primary key |
| `status` | Final status string |
| `message` | Body text that was sent |
| `messageType` | `"imessage"`, `"sms"`, or `"email"` |
| `recipientPhone` / `recipientEmail` / `recipientName` | Where the message went |
| `leadId` | Lead involved (when known) |
| `workspaceId` | Your workspace ID |
| `createdByUserId` | Who created the message |
| `timestamp` | When this event occurred |

## Consuming webhooks safely

<Steps>
  <Step title="Use messageId + event as your key">
    Make updates **idempotent**. If you've already processed a
    `(messageId, event)` pair, ignore duplicates.
  </Step>
  <Step title="Move status forward only">
    Only update your local status in the forward direction:
    `queued â†’ sent â†’ delivered`. Never move backwards.
  </Step>
  <Step title="Return 200 quickly">
    Tuco expects a `2xx` response within a few seconds. Do heavy processing
    asynchronously after acknowledging the webhook.
  </Step>
  <Step title="Log workspaceId and leadId">
    These fields let you trace events back to the right tenant and contact
    in multi-workspace setups.
  </Step>
</Steps>

<Warning>
Tuco may retry webhook delivery if your endpoint returns a non-2xx status.
Always make your handler idempotent.
</Warning>
