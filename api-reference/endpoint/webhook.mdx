## Overview

Webhooks allow you to receive real-time HTTP notifications when specific events occur in your Tuco AI account. Configure webhook endpoints to automate workflows, sync data with external systems, and build custom integrations.

<Info>
**How Webhooks Work:** When an event occurs (like a message being sent), Tuco sends an HTTP POST request to your configured webhook URL with event details.
</Info>

</ParamField>

<ParamField path="lead.updated" type="event">
  Triggered when lead information is modified

  Triggered when availability check completes

  **Payload includes:** Lead ID, old status, new status

<ParamField path="message.sent" type="event">

  **Payload includes:** Message object, recipient, delivery timestamp

  **Payload includes:** Message object, error details, failure reason

  **Payload includes:** Message object, scheduled time

## Webhook Payload Structure

All webhook payloads follow this structure:

```json
{
  "id": "evt_1234567890",
  "type": "message.sent",
  "created": 1634567890,
  "data": {
    // Event-specific data
  },
  "workspace": {
    "id": "ws_xxxxx",
    "name": "Your Workspace"
  }
}
```

### Common Fields

<ResponseField name="id" type="string">
  Unique event identifier (for deduplication)
</ResponseField>

<ResponseField name="type" type="string">
  Event type (see Available Events)
</ResponseField>

<ResponseField name="created" type="integer">
  Unix timestamp when event occurred
</ResponseField>

<ResponseField name="data" type="object">
  Event-specific payload data
</ResponseField>

<ResponseField name="workspace" type="object">
  Your workspace information
</ResponseField>

## Security & Verification

### Webhook Signatures

All webhook requests include a signature in the `X-Tuco-Signature` header for verification.

**Header Format:**

```
X-Tuco-Signature: t=1634567890,v1=a2114d57b48eac39b9ad189dd8316235a7b4a8d21a10bd27519666489c69b503
```

### Verify Signature (Node.js)

```javascript
const crypto = require('crypto');

function verifyWebhookSignature(payload, signature, secret) {
  const elements = signature.split(',');
  const timestamp = elements[0].split('=')[1];
  const receivedSignature = elements[1].split('=')[1];
  
  const signedPayload = `${timestamp}.${JSON.stringify(payload)}`;
  const expectedSignature = crypto
    .createHmac('sha256', secret)
    .update(signedPayload)
    .digest('hex');
  
  return crypto.timingSafeEqual(
    Buffer.from(expectedSignature),
    Buffer.from(receivedSignature)
  );
}

// Usage in Express
app.post('/webhook', (req, res) => {
  const signature = req.headers['x-tuco-signature'];
  const isValid = verifyWebhookSignature(
    req.body,
    signature,
    process.env.WEBHOOK_SECRET
  );
  
  if (!isValid) {
    return res.status(401).send('Invalid signature');
  }
  
  // Process webhook
  console.log('Event:', req.body.type);
  res.status(200).send('OK');
});
```

### Verify Signature (Python)

```python
import hmac
import hashlib
import json

def verify_webhook_signature(payload, signature, secret):
    elements = signature.split(',')
    timestamp = elements[0].split('=')[1]
    received_signature = elements[1].split('=')[1]
    
    signed_payload = f"{timestamp}.{json.dumps(payload)}"
    expected_signature = hmac.new(
        secret.encode(),
        signed_payload.encode(),
        hashlib.sha256
    ).hexdigest()
    
    return hmac.compare_digest(expected_signature, received_signature)

# Usage in Flask
from flask import Flask, request

app = Flask(__name__)

@app.route('/webhook', methods=['POST'])
def webhook():
    signature = request.headers.get('X-Tuco-Signature')
    is_valid = verify_webhook_signature(
        request.json,
        signature,
        os.environ['WEBHOOK_SECRET']
    )
    
    if not is_valid:
        return 'Invalid signature', 401
    
    # Process webhook
    print('Event:', request.json['type'])
    return 'OK', 200
```

## Best Practices

<AccordionGroup>
  <Accordion title="Verify Signatures" icon="shield-check">
    Always verify webhook signatures to ensure requests are from Tuco and haven't been tampered with.
  </Accordion>
  
  <Accordion title="Return 200 Quickly" icon="gauge-high">
    Respond with HTTP 200 within 5 seconds. Process heavy operations asynchronously.
  </Accordion>
  
  <Accordion title="Handle Duplicates" icon="copy">
    Use the event `id` to deduplicate webhook events in case of retries.
  </Accordion>
  
  <Accordion title="Implement Retry Logic" icon="rotate">
    If your endpoint fails, Tuco will retry with exponential backoff up to 3 times.
  </Accordion>
  
  <Accordion title="Use HTTPS" icon="lock">
    Webhook URLs must use HTTPS for security. HTTP endpoints are not supported.
  </Accordion>
  
  <Accordion title="Monitor & Log" icon="chart-line">
    Log all webhook events for debugging and audit purposes.
  </Accordion>
</AccordionGroup>

## Retry Behavior

If your webhook endpoint fails (non-200 response or timeout), Tuco will retry:

<Steps>
  <Step title="Immediate Retry">
    First retry after 1 minute
  </Step>
  
  <Step title="Second Retry">
    Second retry after 5 minutes
  </Step>
  
  <Step title="Final Retry">
    Final retry after 30 minutes
  </Step>
  
  <Step title="Disabled">
    After 3 failed attempts, webhook is automatically disabled
  </Step>
</Steps>

<Warning>
**Webhook Disabled:** You'll receive an email notification if your webhook is disabled due to repeated failures.
</Warning>

## Common Issues

<AccordionGroup>
  <Accordion title="Webhook Not Receiving Events" icon="circle-exclamation">
    **Checks:**
    - Verify webhook is active
    - Ensure URL is accessible from internet
    - Check firewall/security groups allow incoming requests
    - Verify HTTPS certificate is valid
  </Accordion>
  
  <Accordion title="Signature Verification Failing" icon="shield-xmark">
    **Solutions:**
    - Use the correct webhook secret
    - Verify timestamp is within 5 minutes
    - Check payload is unmodified
    - Use raw request body for signature verification
  </Accordion>
  
  <Accordion title="Webhook Disabled Automatically" icon="ban">
    **Reasons:**
    - Endpoint returning non-200 responses
    - Endpoint timing out (>5 seconds)
    - SSL/HTTPS errors
    
    **Fix:** Resolve endpoint issues and re-enable in dashboard
  </Accordion>
</AccordionGroup>
